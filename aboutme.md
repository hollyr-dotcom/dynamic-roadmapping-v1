# About Me

I'm a Senior Staff Designer with 14 years of experience, deeply fluent in design systems, product thinking, and frontend concepts — but not a developer. I can read and tweak basic HTML and CSS, but I rely on you for anything beyond that. I've always paired with engineers, so I understand how things work but don't write production code myself.

## My Work Context

I work at Miro on the Tables team. I design data products — **table, timeline, and kanban views** — used both inside Miro spaces (full screen) and embedded on canvas. My work is for product development teams. I design tools that help them connect strategy to work and plan, track, and execute.

## What I'm Usually Building

**9 times out of 10, I'm prototyping.** My typical flow:

1. Vibe code a working prototype in Claude Code
2. Bring into Figma to refine details and think through the design
3. Return to Claude Code to make it interactive, add motion, and polish the visual design

Prototypes are for demos. They need to **look great and demonstrate complex functionality** — including user input and agentic flows — but the code does not need to be production-ready. Keep it clean enough to reason about, but don't over-engineer.

## Design Aesthetic

- **Minimal with breathing room**: generous white space, clear hierarchy, nothing unnecessary
- **Typography-driven**: let type do the heavy lifting; don't decorate what type can communicate
- **Motion-forward**: use transitions and micro-interactions thoughtfully — they should feel purposeful, not decorative
- **Component-minded**: use repeatable, consistent patterns and components wherever possible; avoid one-off solutions

## Miro Design System

- Use the **Miro Design System MCP** and **Figma** as your primary reference for components, tokens, icons, and patterns
- The design system is the **default, not the rule** — extend and adapt it when the prototype calls for it
- When you deviate, tell me what you changed and why
- Always prefer design system tokens (colours, spacing, type scales) over hardcoded values, even when extending

## How to Talk to Me

- Plain English, no jargon
- When you change code, tell me **what** changed and **why** — briefly
- If something might break or is a meaningful tradeoff, flag it before doing it
- Be conversational, not mechanical — "Here's the table interaction, give it a try" not a bullet-by-bullet technical log
- Don't repeat in text what the UI already shows

## Tech Preferences

- **Bun** over npm for everything
- **Tailwind CSS** for styling
- Mobile-friendly by default
- Keep code simple and readable — these are demos

## Git

- Commit working features immediately with clear, descriptive messages
- Never commit broken code
- If something is in progress, say so before committing a partial state

## Testing Prototypes

- Actually interact with things — click, type, trigger flows
- For canvas or data-heavy elements, test with realistic content
- Take screenshots and analyse them
- Tell me what's broken before fixing it
